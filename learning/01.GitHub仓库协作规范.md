## GitHub 仓库协作规范

### 学习目标
- 理解为什么需要协作规范
- 学会使用 Git 分支进行开发
- 掌握 Pull Request（PR）流程
- 了解 Commit 消息规范
- 学会解决合并冲突
- 掌握团队协作的最佳实践

### 前置要求
- 已完成课程 00：GitHub 协作基础
- 能够熟练使用 `git clone`、`git add`、`git commit`、`git push`、`git pull`

### 路线步骤

#### 步骤 1：理解为什么需要协作规范

在开始学习具体操作之前，先了解为什么团队协作需要规范。

<!-- 用通俗的语言向学员解释：

想象一下，如果团队里每个人都按照自己的方式写代码、提交代码，会发生什么？
- 有人直接在 main 分支上改代码，导致项目崩溃
- 有人提交的代码没有说明，别人看不懂改了什么
- 两个人同时改了同一个文件，代码互相覆盖
- 出了问题不知道是谁、什么时候改的

协作规范就是为了解决这些问题，让团队成员能够：
1. 安全地开发新功能（不影响正在运行的代码）
2. 清楚地知道每个人改了什么
3. 在代码合并前进行检查
4. 出现问题时能够快速定位和回退

告诉学员：本课程将学习团队中最常用的协作流程——"分支 + Pull Request"模式。

【检查点】确认学员能回答：
- 为什么不能所有人都直接在 main 分支上改代码？
- 协作规范能解决什么问题？
-->

#### 步骤 2：分支（Branch）基础

##### 2.1 什么是分支

<!-- 用比喻向学员解释：

分支就像是游戏的"存档点"。main 分支是主线剧情，当你想尝试新功能或实验性改动时，可以创建一个新的分支（相当于新建一个存档），在这个分支上随便改，不会影响主线。

如果改动成功了，就把这个分支合并回 main（相当于把新存档的内容合并到主线）；
如果改砸了，直接删除这个分支就行，main 分支完全不受影响。

技术定义（简单了解即可）：
- 分支是 Git 中的一个轻量级可移动指针，指向某个提交
- 创建分支只是创建了一个新指针，非常快速
- 切换分支时，Git 会更新工作目录的文件内容

常用分支：
- main（或 master）：主分支，始终保持可部署的稳定状态
- develop：开发分支（有些团队使用）
- feature/xxx：功能分支，用于开发新功能
- fix/xxx：修复分支，用于修复 bug
-->

##### 2.2 查看和创建分支

1. 查看当前所在分支
  <!-- 让学员在 Team 仓库目录下执行：
    ```bash
    git branch
    ```
    预期输出：
    ```
    * main
    ```
    星号（*）标记的就是当前所在分支。

    如果还想看远程分支，执行：
    ```bash
    git branch -a
    ```
    会显示本地和远程的所有分支。
  -->

2. 创建新分支
  <!-- 让学员创建一个练习分支：
    ```bash
    git branch feature/practice
    ```
    这条命令创建了一个名为 feature/practice 的新分支，但不会自动切换过去。

    命名规范：
    - 功能分支：feature/功能名称（如 feature/login、feature/user-profile）
    - 修复分支：fix/问题描述（如 fix/login-bug、fix/typo）
    - 使用小写字母和连字符，不要用空格或特殊字符

    告诉学员：分支命名要清晰，让别人一看就知道这个分支是做什么的。
  -->

3. 切换到新分支
  <!-- 让学员执行：
    ```bash
    git checkout feature/practice
    ```
    预期输出：
    ```
    Switched to a new branch 'feature/practice'
    ```

    或者使用一条命令完成创建+切换：
    ```bash
    git checkout -b feature/practice
    ```
    -b 参数表示"创建并切换"。

    验证切换成功：
    ```bash
    git branch
    ```
    应该看到：
    ```
    * feature/practice
      main
    ```

    告诉学员：现在所有修改都在 feature/practice 分支上，不会影响 main 分支。
  -->

4. 在新分支上进行修改
  <!-- 让学员在 feature/practice 分支上做一些修改来练习：

    1. 创建一个新文件：
    ```bash
    echo "# 练习分支" > practice.md
    ```
    或者用编辑器创建 practice.md，写入一些内容。

    2. 提交修改：
    ```bash
    git add practice.md
    git commit -m "添加练习文件"
    ```

    3. 查看提交历史：
    ```bash
    git log --oneline
    ```
    预期输出类似：
    ```
    abc1234 添加练习文件
    def5678 之前的某个提交
    ...
    ```

    告诉学员：这个提交只存在于 feature/practice 分支，main 分支看不到这个修改。
  -->

5. 切换回 main 分支
  <!-- 让学员执行：
    ```bash
    git checkout main
    ```
    然后查看文件：
    ```bash
    ls
    ```
    或者在 Windows 上：
    ```bash
    dir
    ```
    学员会发现 practice.md 文件"消失"了。

    解释：不是文件被删除了，而是这个文件只存在于 feature/practice 分支。切换回 main 后，工作目录更新为 main 分支的内容，所以看不到 practice.md。

    让学员切换回 feature/practice 验证：
    ```bash
    git checkout feature/practice
    ls
    ```
    practice.md 又出现了。

    【检查点】确认学员能回答：
    - 分支的作用是什么？
    - 如何在分支之间切换？
    - 为什么切换分支后文件会"消失"又"出现"？
  -->

#### 步骤 3：Pull Request（PR）流程

##### 3.1 什么是 Pull Request

<!-- 向学员解释：

Pull Request（拉取请求，简称 PR）是 GitHub 上用于代码审查和讨论的机制。

工作流程：
1. 你在自己的分支上完成开发
2. 把分支推送到 GitHub
3. 创建一个 PR，请求把你的分支合并到 main
4. 团队成员审查你的代码，提出意见
5. 审查通过后，合并 PR

为什么需要 PR？
- 代码审查：在合并前让其他人检查代码质量
- 讨论交流：可以在 PR 中讨论代码实现
- 自动测试：可以配置自动化测试，确保代码没有问题
- 记录历史：PR 本身就是详细的变更记录

告诉学员：PR 是团队协作的核心流程，几乎所有开源项目和团队都使用这个模式。
-->

##### 3.2 推送分支到远程

1. 推送本地分支到 GitHub
  <!-- 让学员确保在 feature/practice 分支上：
    ```bash
    git checkout feature/practice
    ```

    然后推送分支：
    ```bash
    git push origin feature/practice
    ```
    预期输出：
    ```
    * [new branch]      feature/practice -> feature/practice
    ```

    解释命令：
    - origin：远程仓库的默认名称
    - feature/practice：要推送的分支名

    告诉学员：现在 GitHub 上也有了这个分支，可以创建 PR 了。

    常见问题：
    - 如果提示 "fatal: The current branch ... has no upstream branch"
      说明这是第一次推送这个分支，使用：
      ```bash
      git push -u origin feature/practice
      ```
      -u 参数会建立本地分支和远程分支的关联，以后直接 git push 就可以了。
  -->

##### 3.3 创建 Pull Request

1. 在 GitHub 上创建 PR
  <!-- 引导学员操作：

    1. 打开 Team 仓库页面：https://github.com/Uxiyu-Team/Team
    2. GitHub 通常会自动检测到新推送的分支，并显示黄色提示框：
       "feature/practice had recent commits"
       点击 "Compare & pull request" 按钮
    3. 如果没有看到提示，手动创建：
       - 点击 "Pull requests" 标签
       - 点击 "New pull request" 按钮
       - 在 "compare" 下拉框选择 feature/practice 分支
       - 确保 "base" 是 main 分支

    4. 填写 PR 信息：
       - Title（标题）：简明描述这次改动，如"添加练习文件"
       - Description（描述）：详细说明改了什么、为什么改
         可以使用模板：
         ```
         ## 改动内容
         - 添加了 practice.md 文件

         ## 改动原因
         - 用于练习分支和 PR 流程

         ## 测试情况
         - 已在本地测试，文件可以正常创建和读取
         ```

    5. 点击 "Create pull request" 按钮

    告诉学员：PR 创建后，团队成员会收到通知，可以开始审查代码了。
  -->

2. 查看 PR 页面
  <!-- 让学员观察 PR 页面的各个部分：

    - 标题和描述：你刚才填写的内容
    - Commits 标签：这个分支的所有提交
    - Files changed 标签：所有修改的文件，可以看到具体的代码变更
      - 绿色背景：新增的行
      - 红色背景：删除的行
    - 审查区域：团队成员可以在这里添加评论

    告诉学员：审查者会主要看 "Files changed" 部分，检查代码改动是否合理。
  -->

##### 3.4 代码审查（Code Review）

<!-- 向学员解释代码审查的流程：

代码审查是 PR 流程中最重要的环节，目的是：
1. 发现潜在的问题和 bug
2. 确保代码符合团队规范
3. 知识共享：让团队成员了解彼此的工作
4. 提升代码质量

审查者的工作：
1. 查看 "Files changed" 中的代码变更
2. 点击具体行号，添加评论
3. 提出建议或询问问题
4. 如果没问题，点击 "Approve" 批准
5. 如果有问题，点击 "Request changes" 要求修改

PR 作者的工作：
1. 查看审查意见
2. 在本地修改代码
3. 提交并推送到同一个分支（PR 会自动更新）
4. 回复审查意见，说明已修改

告诉学员：代码审查不是找茬，而是帮助提升代码质量。要虚心接受意见，也要敢于提出自己的看法。

【检查点】确认学员能回答：
- PR 的作用是什么？
- 代码审查的目的是什么？
- 如果审查者提出修改意见，应该怎么做？
-->

##### 3.5 合并 Pull Request

1. 合并 PR 到 main 分支
  <!-- 引导学员操作：

    假设 PR 已经通过审查（在实际团队中需要等待审查通过），现在合并 PR：

    1. 在 PR 页面底部，点击 "Merge pull request" 按钮
    2. 确认合并信息，再次点击 "Confirm merge"
    3. 合并成功后，会显示 "Merged" 状态

    告诉学员：
    - 合并后，feature/practice 分支的改动已经应用到 main 分支
    - 可以选择删除远程分支（GitHub 会提示），通常合并后就可以删除了
    - 本地分支也可以删除（下一步会讲）

    注意：在实际团队中，合并权限可能有限制，需要管理员批准。这里只是让学员了解流程。
  -->

2. 同步本地 main 分支
  <!-- 让学员执行：
    ```bash
    git checkout main
    git pull
    ```
    解释：
    - checkout main：切换到 main 分支
    - pull：从远程拉取最新内容（包括刚才合并的 PR）

    让学员查看文件：
    ```bash
    ls
    ```
    practice.md 文件现在出现在 main 分支了！

    告诉学员：这就是 PR 合并的效果——把分支的改动应用到 main。
  -->

3. 清理分支
  <!-- 让学员删除本地和远程的 feature/practice 分支：

    删除本地分支：
    ```bash
    git branch -d feature/practice
    ```
    预期输出：
    ```
    Deleted branch feature/practice (was abc1234).
    ```

    如果远程分支还没删除（在 GitHub 上合并时没有勾选删除），可以手动删除：
    ```bash
    git push origin --delete feature/practice
    ```

    告诉学员：
    - 合并后的分支已经没有用了，及时清理可以保持仓库整洁
    - -d 参数表示"删除"，Git 会检查分支是否已合并，如果未合并会警告
    - 如果确定要删除未合并的分支，使用 -D（大写）参数，但要谨慎

    【检查点】确认学员能回答：
    - 如何创建 PR？
    - PR 合并后，分支的改动去了哪里？
    - 为什么要删除已合并的分支？
  -->

#### 步骤 4：Commit 消息规范

##### 4.1 为什么需要规范的 Commit 消息

<!-- 向学员解释：

Commit 消息是代码历史记录的重要组成部分。好的 commit 消息可以：
1. 快速了解每次提交改了什么
2. 在出现问题时快速定位
3. 生成变更日志（Changelog）
4. 方便团队协作和代码审查

不好的 commit 消息：
- "fix" —— 修了什么？
- "update" —— 更新了什么？
- "111" —— 完全看不懂
- "asdfasdf" —— 无意义

好的 commit 消息应该：
- 简明扼要地描述改动内容
- 说明"为什么"而不仅仅是"做了什么"
- 遵循团队统一的格式

告诉学员：团队通常会制定 commit 规范，下面介绍一种常用的规范。
-->

##### 4.2 Conventional Commits 规范

<!-- 向学员介绍 Conventional Commits（约定式提交）规范：

格式：
```
<类型>: <简短描述>

<详细说明（可选）>
```

常用类型：
- feat: 新功能（feature）
- fix: 修复 bug
- docs: 文档改动（documentation）
- style: 代码格式改动（不影响代码运行）
- refactor: 代码重构（既不是新功能也不是修复 bug）
- test: 测试相关
- chore: 构建过程或辅助工具的改动

示例：
```
feat: 添加用户登录功能

- 实现用户名密码登录
- 添加登录表单验证
- 集成 JWT token 认证
```

```
fix: 修复登录页面在 Safari 浏览器上的显示问题

Safari 不支持 flexbox 的某些特性，改用 grid 布局。
```

```
docs: 更新 README 中的安装说明

添加了对 Windows 系统的安装步骤说明。
```

告诉学员：
- 第一行（简短描述）不超过 50 个字符
- 使用祈使语气（"添加"而不是"添加了"）
- 第一行不需要句号结尾
- 详细说明（可选）解释"为什么"和"怎么做"

让学员练习：为之前的提交写一个规范的 commit 消息。
-->

##### 4.3 修改 Commit 消息

1. 修改最近一次的 commit 消息
  <!-- 让学员练习修改 commit 消息：

    如果刚提交完就发现消息写错了，可以修改：
    ```bash
    git commit --amend -m "新的 commit 消息"
    ```

    例如：
    ```bash
    git commit --amend -m "docs: 添加练习文件用于学习分支和 PR 流程"
    ```

    告诉学员：
    - --amend 参数表示"修改上一次提交"
    - 这会替换上一次的 commit 消息
    - 如果还没有 push，这是安全的
    - 如果已经 push，修改 commit 后需要 force push（强制推送），但要谨慎，因为会影响其他人

    警告：不要修改已经 push 且其他人已经拉取的 commit，这会导致混乱。
  -->

2. 查看 commit 历史
  <!-- 让学员查看 commit 历史：
    ```bash
    git log --oneline
    ```
    预期输出：
    ```
    abc1234 docs: 添加练习文件用于学习分支和 PR 流程
    def5678 之前的某个提交
    ...
    ```

    其他有用的命令：
    ```bash
    git log --oneline -10
    ```
    只显示最近 10 条提交。

    ```bash
    git log --oneline --graph
    ```
    以图形方式显示分支历史。

    告诉学员：经常查看 commit 历史，可以帮助了解项目的演进过程。

    【检查点】确认学员能回答：
    - 好的 commit 消息应该包含什么？
    - feat: 和 fix: 有什么区别？
    - 如何修改最近一次的 commit 消息？
  -->

#### 步骤 5：解决合并冲突

##### 5.1 什么是合并冲突

<!-- 向学员解释：

合并冲突发生在：两个人（或两个分支）同时修改了同一个文件的同一部分，Git 无法自动判断应该保留哪个版本。

例如：
- 你在 main 分支的第 10 行添加了代码 A
- 同事在 feature 分支的第 10 行添加了代码 B
- 当你想把 feature 合并到 main 时，Git 不知道应该保留 A 还是 B

冲突不是错误，而是正常的协作情况。Git 会标记冲突的位置，让你手动决定如何合并。

告诉学员：遇到冲突不要慌，这是团队协作中的常见情况。
-->

##### 5.2 模拟合并冲突

1. 创建冲突场景
  <!-- 让学员创建一个冲突场景来练习：

    1. 确保在 main 分支上：
    ```bash
    git checkout main
    ```

    2. 创建一个新的练习分支：
    ```bash
    git checkout -b feature/conflict-practice
    ```

    3. 创建一个文件 test.txt：
    ```bash
    echo "第一行" > test.txt
    echo "第二行" >> test.txt
    echo "第三行" >> test.txt
    ```
    或者用编辑器创建 test.txt，内容为：
    ```
    第一行
    第二行
    第三行
    ```

    4. 提交：
    ```bash
    git add test.txt
    git commit -m "添加测试文件"
    ```

    5. 切换到 main 分支：
    ```bash
    git checkout main
    ```

    6. 修改 test.txt 的第二行：
    用编辑器打开 test.txt，把"第二行"改为"main 分支修改的第二行"

    7. 提交：
    ```bash
    git add test.txt
    git commit -m "修改 main 分支的第二行"
    ```

    8. 切换回 feature 分支：
    ```bash
    git checkout feature/conflict-practice
    ```

    9. 也修改 test.txt 的第二行：
    用编辑器打开 test.txt，把"第二行"改为"feature 分支修改的第二行"

    10. 提交：
    ```bash
    git add test.txt
    git commit -m "修改 feature 分支的第二行"
    ```

    告诉学员：现在两个分支都修改了同一个文件的同一行，合并时会产生冲突。
  -->

2. 触发合并冲突
  <!-- 让学员尝试合并：
    ```bash
    git checkout main
    git merge feature/conflict-practice
    ```
    预期输出：
    ```
    Auto-merging test.txt
    CONFLICT (content): Merge conflict in test.txt
    Automatic merge failed; fix conflicts and then commit the result.
    ```

    告诉学员：Git 提示有冲突，需要手动解决。
  -->

##### 5.3 解决合并冲突

1. 查看冲突文件
  <!-- 让学员用编辑器打开 test.txt，会看到类似这样的内容：
    ```
    第一行
    <<<<<<< HEAD
    main 分支修改的第二行
    =======
    feature 分支修改的第二行
    >>>>>>> feature/conflict-practice
    第三行
    ```

    解释冲突标记：
    - <<<<<<< HEAD：当前分支（main）的内容开始
    - =======：分隔符
    - >>>>>>> feature/conflict-practice：要合并的分支的内容结束

    告诉学员：Git 标记了冲突的位置，你需要决定保留哪个版本，或者合并两个版本。
  -->

2. 手动解决冲突
  <!-- 让学员用编辑器修改 test.txt，决定最终的内容。

    方案 1：保留 main 分支的版本
    ```
    第一行
    main 分支修改的第二行
    第三行
    ```

    方案 2：保留 feature 分支的版本
    ```
    第一行
    feature 分支修改的第二行
    第三行
    ```

    方案 3：合并两个版本
    ```
    第一行
    ```

### 相关问题

- **什么时候应该创建新分支？**
  > 每次开发新功能、修复 bug、或进行实验性改动时，都应该创建新分支。不要在 main 分支上直接开发。

- **PR 太大怎么办？**
  > PR 应该小而专注。如果一个 PR 改动太多，考虑拆分成多个 PR。每个 PR 只解决一个问题或实现一个功能。

- **冲突太多了，怎么办？**
  > 频繁从 main 分支同步代码，可以大大减少冲突。如果冲突已经很多，耐心一个一个解决，不要急。

- **可以修改已经 push 的 commit 吗？**
  > 技术上可以（使用 force push），但不推荐，特别是其他人已经拉取了这些 commit。这会打乱团队的开发流程。如果必须修改，先和团队沟通。

- **main 分支被破坏了怎么办？**
  > 使用 git revert 或 git reset 回退到之前的稳定版本。git revert 会创建一个新的提交来撤销改动（安全），git reset 会直接删除提交（危险，要谨慎使用）。
