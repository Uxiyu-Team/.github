# AGENTS.md - AI 维护指南

本文档面向后续接手维护 `learning/` 目录的 AI 助手，说明如何维护这些教学内容以及需要遵守的安全边界。

## 目录结构说明

```
learning/
├── README.md          # 学习路线总纲（面向人类和 AI）
├── SKILL.md           # AI 教学技能定义（面向 AI）
├── 00.GitHub团队协作基础.md  # 课程 0（面向人类和 AI）
├── 01.GitHub仓库协作规范.md  # 课程 1（面向人类和 AI）
└── AGENTS.md          # 本文档（面向 AI 维护者）
```

## 文档设计原则

### 双层内容结构

所有课程文档采用**双层结构**：
- **可见内容**：面向人类学员的学习目标和步骤清单
- **注释内容**（HTML 注释 `<!-- -->`）：面向 AI 的教学提示、参考答案、检查点、常见问题

**维护时必须同时维护两层内容**：
1. 可见内容保持简洁清晰，适合新手阅读
2. 注释内容提供详细的教学指导，帮助 AI 正确教学

### 文件职责划分

| 文件 | 主要受众 | 职责 |
|------|---------|------|
| README.md | 人类学员 + AI | 学习路线总纲、使用教程、课程目录 |
| SKILL.md | AI 工具 | AI 教学技能定义、工作流程、话术示例 |
| 课程文件 | 人类学员 + AI | 具体课程的学习目标、步骤、注释 |
| AGENTS.md | AI 维护者 | 维护指南、安全边界（本文件） |

## 维护指南

### 添加新课程

1. 创建新文件，命名格式：`<编号>.<课程名称>.md`（如 `02.Git分支与合并.md`）
2. 参考 `00.GitHub团队协作基础.md` 的结构：
   - 学习目标
   - 路线步骤（每个步骤包含子步骤和注释）
   - 检查点（在注释中标记 `【检查点】`）
   - 完成标志
   - 常见问题
3. 在 `README.md` 的课程目录表格中添加新课程的条目
4. 在 `SKILL.md` 的「文件读取说明」表格中添加新文件的 raw 链接

### 修改现有课程

1. **修改可见内容**时：
   - 保持语言简洁、适合新手
   - 确保步骤顺序合理、不跳步
   - 命令示例必须准确可执行

2. **修改注释内容**时：
   - 确保教学提示清晰、具体
   - 命令示例的预期输出必须准确
   - 常见问题应覆盖新手可能遇到的典型错误
   - 检查点问题应能验证学员是否真正理解

3. **修改后检查**：
   - 通过 raw 链接查看文件，确认注释内容完整
   - 测试所有命令示例是否可执行
   - 确保链接（如 GitHub 仓库链接）仍然有效

### 更新 SKILL.md

如果教学流程或话术需要调整：
1. 确保「文件读取说明」表格包含所有课程文件的 raw 链接
2. 更新「教学话术示例」以反映新的教学方法
3. 确保「工作流程」描述清晰、可操作

## 安全边界

### 不应该做的事情

1. **不要删除或修改双层结构**
   - 不要移除 HTML 注释中的教学提示
   - 不要把注释内容变成可见内容（注释是给 AI 的，不是给学员的）

2. **不要破坏教学流程**
   - 不要跳步或重新排序步骤（除非有充分理由并更新所有相关内容）
   - 不要删除检查点（它们是验证学员理解的关键）

3. **不要引入未经验证的内容**
   - 所有命令示例必须经过测试，确保可执行
   - 所有链接必须经过验证，确保有效
   - 不要添加可能过时的信息（如具体版本号，除非必要）

4. **不要修改安全相关的内容**
   - 不要修改涉及认证、权限、token 的说明（除非确认新的方法更安全）
   - 不要建议学员禁用安全特性（如 SSL 验证）

5. **不要改变文件职责**
   - 不要把 SKILL.md 的内容放到课程文件中
   - 不要把课程的具体内容放到 README.md 中

### 应该做的事情

1. **保持一致性**
   - 新课程的格式应与现有课程保持一致
   - 术语使用应统一（如统一使用"仓库"而非"项目"）

2. **考虑新手视角**
   - 假设学员是零基础，不要假设他们知道任何前置知识
   - 用通俗的语言解释概念，避免堆砌术语
   - 提供足够的上下文和示例

3. **考虑 AI 教学需求**
   - 注释中的教学提示应该具体、可操作
   - 提供常见错误及其解决方案
   - 设计有效的检查点问题

4. **测试和验证**
   - 修改后通过 raw 链接查看文件，确认内容完整
   - 测试所有命令示例
   - 验证所有链接

5. **更新相关文档**
   - 添加新课程时，更新 README.md 和 SKILL.md
   - 修改教学流程时，更新 SKILL.md
   - 保持文档之间的链接有效

## 版本控制

### 提交信息规范

- 添加新课程：`feat(learning): 添加课程 02 - <课程名称>`
- 修改课程：`fix(learning): 修正课程 01 中的 <具体内容>`
- 更新文档：`docs(learning): 更新 <文件名> 的 <具体内容>`

### 不提交的内容

`AGENTS.md` 本身不应该提交到 git（已在 `.gitignore` 中配置），因为：
- 它是给 AI 维护者看的，不是给学员看的
- 可能包含敏感信息（如内部维护流程）
- 不同 AI 助手可能有不同的维护需求

## 维护经验总结

在维护 `learning/` 目录时，积累了一些最佳实践和注意事项：

### 文件命名规范

- 课程文件使用两位十进制数索引（含前导0），如 `00.xxx.md`、`01.xxx.md`
- 这样可以保证文件按正确顺序排序，避免 `10.xxx.md` 排在 `2.xxx.md` 前面的问题
- 新增课程应延续此模式，从 `02.` 开始

### 内容组织原则

- **双层结构**是核心：可见内容给学员，注释内容给 AI
- 所有教学提示、检查点、常见问题都放在 HTML 注释中
- 不要为了方便而把注释内容变成可见内容

### 修改流程

1. **先重命名文件**：使用 `move_file` 工具重命名，确保文件系统更新
2. **再更新引用**：在所有相关文件中更新链接和路径
3. **最后验证**：通过 raw 链接查看文件，确认所有链接正常

### 注意事项

- 更新 SKILL.md 中的「文件读取说明」表格时，必须包含所有课程文件的 raw 链接
- AGENTS.md 是给 AI 维护者看的内部文档，不提交到版本控制
- 每次修改后都要检查 README.md 的课程目录是否同步更新

### 常见陷阱

- 忘记更新 .gitignore：如果新增了不应提交的文件，记得添加忽略规则
- 字符编码问题：Windows 环境下中文文件名可能显示异常，优先使用 move_file 而不是 shell 命令
- 引用遗漏：重命名文件后，容易遗漏某些文件中的引用，建议用 grep 全局搜索旧文件名
